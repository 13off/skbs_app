# Архитектура AppСтрой

## Общая схема

```text
Flutter web/PWA, Android, desktop
        │
        ├── Supabase Auth
        ├── PostgreSQL + RLS + RPC
        ├── Storage + signed/download access
        ├── Realtime broadcast
        └── JWT-защищённые Edge Functions
```

Flutter-клиент отвечает за представление, редактируемые черновики, локальную подготовку документов и явное подтверждение пользователя. Сервер отвечает за идентичность, компанию, роль, объект, целостность данных и окончательное разрешение операции.

## Слои Flutter

- `lib/app` — тема и общие настройки приложения;
- `lib/models` — совместимые публичные экспорты моделей;
- `lib/data` — общие репозитории, кеши и синхронизация;
- `lib/features` — функциональные модули по предметным областям;
- `lib/screens` — публичные маршруты и совместимые входы;
- `lib/widgets` — переиспользуемые элементы интерфейса;
- `lib/services` — платформенные и инфраструктурные сервисы.

Крупные экраны разделяются на состояние, загрузку, действия, секции и представление. Совместимые публичные классы и маршруты сохраняются, чтобы рефакторинг не менял поведение установленного клиента.

## Компании, люди и объекты

- `company_id` разделяет данные арендаторов;
- `object_id` является стабильной связью с объектом;
- `person_id` объединяет назначения одного физического человека на разных объектах;
- `employee_id` остаётся идентификатором конкретного назначения;
- старые текстовые поля объекта сохраняются там, где нужны для обратной совместимости.

Однофамильцы не объединяются автоматически. Копирование назначения сохраняет `person_id`, а ставка, должность, статус и объект остаются свойствами назначения.

## Роли и платформы

`AuthGate` загружает сессию и профиль. `MainScreen` выбирает отдельную платформу роли:

- developer;
- admin/manager;
- foreman;
- hr;
- accountant;
- lawyer;
- базовая рабочая оболочка.

Скрытие кнопки не является проверкой права. Репозиторий, RLS, RPC или Edge Function обязаны повторно проверить доступ.

## Кеши и Realtime

`AppDataSync` принимает серверные broadcast-события и объединяет близкие изменения. `AppCacheCoordinator` переводит предметные домены в конкретные кеши и очищает только зависимые области.

При смене пользователя или компании очищаются все пользовательские кеши. Несохранённый локальный табель не должен перезаписываться realtime-событием.

## ИИ-помощник

ИИ-серверы подготовки являются read-only. Они возвращают типизированное предложение, а не выполняют запись.

Жизненный цикл действия:

```text
proposed → confirmed → completed
         ↘ cancelled
         ↘ failed
```

Все действия проходят единый координатор и записываются в `ai_action_audit`. Создание сотрудников, выплат, задач и документов возвращает пользователя в обычную форму. Прямая запись допускается только для явно предусмотренных операций после отдельного подтверждения.

## Документы

- каталог хранит шаблоны и версии;
- закрытые исходники находятся в Storage или подключённом рабочем источнике;
- точные DOCX формируются локально;
- кадровые и банковские данные читаются через существующие RLS-репозитории;
- содержимое персональных файлов не передаётся Edge Function ИИ;
- ZIP кандидата собирается локально после повторной проверки RLS и согласия.

## Совместимость

Миграции должны учитывать одновременно:

- текущую PWA;
- ранее установленную PWA;
- мобильный релиз;
- временно не обновившиеся устройства.

Удаление или переименование поля выполняется только после переходного периода. Новые обязательные связи сначала заполняются для существующих строк, затем переводятся в `NOT NULL`.
