# Edge Functions

## Общие требования

- функции пользователя разворачиваются с `verify_jwt=true`;
- входящий JWT проверяется через `auth.getUser()`;
- запрос связывается с активной компанией пользователя;
- членство и роль проверяются повторно;
- `service_role` не используется в read-only функциях ИИ;
- ответы с персональными данными минимизируются;
- ошибки не должны раскрывать секреты, SQL или пути закрытого Storage.

Исключения без JWT допустимы только для webhook, landing или job-функций с собственным проверяемым секретом и ограниченным назначением.

## Основные группы

### Приглашения и компании

- создание и обработка приглашений;
- безопасная landing-страница;
- принятие приглашения после явного действия пользователя.

### Push и устройства

- регистрация устройства;
- отправка push;
- выполнение очереди уведомлений;
- публичная выдача только открытого web-push ключа.

### Подбор и Telegram

- webhook Telegram;
- загрузка файлов кандидатов;
- действия HR;
- служебная настройка бота.

Webhook-функции должны проверять собственный секрет или заранее установленный маршрут и не доверять полям сообщения как данным авторизации.

### ИИ

- `ai-assistant` — текстовый помощник;
- `ai-search` — поиск по разрешённым рабочим данным;
- `ai-action-draft` — read-only черновик задачи;
- `ai-document-draft` — read-only подготовка документа;
- `ai-operational-draft` — типизированные операционные предложения.

`ai-operational-draft` проверяет пользователя, компанию, членство, роль и объект. Он не записывает рабочие сущности. Содержимое документов кандидата и Storage-пути не входят в ответ; ZIP формируется локально после RLS.

## Деплой функции

В deploy передаётся полный набор файлов:

```text
index.ts
shared.ts
people_actions.ts
report_actions.ts
...другие относительные зависимости
```

После deploy обязательно проверить:

1. статус `ACTIVE`;
2. ожидаемую версию;
3. `verify_jwt`;
4. отсутствие ошибок импорта;
5. ответ без токена;
6. ответ разрешённой роли;
7. отказ запрещённой роли;
8. логи после первого production-вызова.

## Изменения контракта

Новые поля добавляются совместимо. Старые ключи ответа не переименовываются, пока ими может пользоваться установленная PWA или мобильная версия. Если поведение меняется, добавляется contract-тест Flutter и Deno-проверка функции.
