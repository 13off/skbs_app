# Приёмка и готовность AppСтрой к эксплуатации и показу

## Уровни готовности

### 1. Техническая готовность

Подтверждается автоматическими проверками:

- Flutter analyzer;
- regression tests;
- release web-build;
- backend migration guard;
- Deno check изменённых Edge Functions;
- post-deploy smoke публичной PWA;
- совпадение публичного source marker с merge-коммитом.

### 2. Ролевая готовность

Проверяется в разделе разработчика **Система → Ролевая приёмка**.

Для каждой профессии фиксируются:

- фактическая серверная роль текущего входа;
- активная компания;
- обязательные permissions;
- запрещённые permissions;
- минимальный Data API SELECT через пользовательский JWT и RLS;
- назначенный объект прораба.

Клиентский режим просмотра роли не считается приёмкой, потому что он не меняет JWT, членство и серверную идентичность. Для окончательного результата требуется отдельная тестовая учётная запись каждой профессии.

### 3. Операционная готовность

Перед ежедневной эксплуатацией необходимо вручную пройти на тестовых данных:

1. кандидат → сотрудник → кадровый ZIP → подписи → выход 8/8;
2. задача прораба с действующими ограничениями объекта;
3. дневной и месячный табель;
4. выплата и чек;
5. единый контроль табеля и выплат;
6. уведомления прорабу, бухгалтеру и руководителю.

### 4. Готовность персональных данных

Реальные документы разрешаются только при открытом runtime production gate. Gate требует:

- утверждённый профиль работодателя;
- юридически утверждённые формы;
- российское место хранения;
- правила хранения и удаления;
- журнал доступа;
- проверку резервного копирования и восстановления;
- проверку отзыва доступа;
- назначенного ответственного за инциденты.

До открытия gate используются только тестовые, обезличенные и явно маркированные копии.

### 5. Коммерческая готовность

Минимальный набор перед первым показом клиенту:

- демонстрационный центр внутри приложения;
- отдельная тестовая компания;
- сценарий `docs/commercial-demo.md`;
- позиционирование и целевой сегмент;
- тарифы и ограничения планов;
- лендинг или короткая презентация;
- инструкция первого запуска компании;
- канал поддержки и порядок обработки инцидентов.

## Критерий «можно использовать ежедневно»

Система считается операционно принятой, когда:

- технический CI и post-deploy smoke зелёные;
- каждая фактическая роль прошла live-проверку;
- тестовый кадровый маршрут завершён без повторного ввода;
- единый аудит открывается из бухгалтерской профессии и ничего не изменяет;
- ограничения объекта проверены на тестовой задаче;
- ни один тест не требует открытия real personal-data gate.

## Критерий «можно показывать клиенту»

- используется только отдельная тестовая компания;
- на экране нет персональных или финансовых данных действующей компании;
- демонстрация укладывается в 15 минут;
- каждый сценарий заканчивается измеримым бизнес-результатом;
- ограничения продукта и ручные подтверждения проговариваются честно.
