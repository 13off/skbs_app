# Gate промышленной работы с персональными документами

Статус на момент создания документа: **BLOCKED**.

Реальные паспорта, СНИЛС, ИНН, банковские реквизиты и другие кадровые документы нельзя считать разрешёнными для промышленной загрузки только потому, что технический экран и хранилище уже работают. До отдельного утверждения используются тестовые, обезличенные или явно маркированные тестовые копии.

Машиночитаемый статус находится в `config/personal-data-production-gate.json`. CI не должен позволять включить `real_personal_documents_enabled`, пока не закрыты все обязательные доказательства.

## Обязательные доказательства

Каждый пункт закрывается не устным подтверждением, а ссылкой на документ, настройку, протокол проверки или ответственное лицо.

| Проверка | Что должно быть подтверждено | Текущий статус |
|---|---|---|
| Российское хранение | регион фактического хранения основной базы, Storage, резервных копий и журналов; перечень подрядчиков | Не подтверждено |
| Оператор данных | полное наименование, адрес, ИНН/ОГРН, контакты ответственного | Не подтверждено |
| Согласие | утверждённая юристом форма, цели, состав данных, срок, порядок отзыва и передачи | Не подтверждено |
| Сроки хранения и удаления | сроки по каждому типу документа, основания продления, процедура удаления и актирование | Не подтверждено |
| Журнал доступа | кто открыл, скачал, изменил или удалил документ; дата, компания, кандидат/сотрудник, результат | Не подтверждено |
| Резервное копирование | расписание, шифрование, расположение, срок хранения и успешный тест восстановления | Не подтверждено |
| Отключение доступа | проверка увольнения/смены роли, отзыв сессий, ключей, приглашений и доступа к архивам | Не подтверждено |
| Инциденты | ответственный, канал сообщения, порядок изоляции, расследования и уведомления | Не подтверждено |

## Правило включения

`real_personal_documents_enabled` можно изменить на `true` только когда одновременно выполнены условия:

1. все поля раздела `evidence` равны `true`;
2. указаны `approved_by` и `approved_at`;
3. доказательства приложены к внутреннему решению о запуске;
4. release checklist выполнен на production;
5. проверены RLS, Storage policies, журналы скачивания, резервное копирование и отключение доступа;
6. формы согласия и трудового договора утверждены ответственными лицами.

До этого момента:

- использовать только тестовые, обезличенные или специально маркированные копии;
- не загружать реальные паспортные развороты и банковские реквизиты ради демонстрации;
- не пересылать закрытые документы в ИИ, логи, аналитику, Slack, GitHub или публичные ссылки;
- не заявлять, что контур соответствует требованиям законодательства;
- не удалять существующие записи автоматически и не менять текущие процессы без отдельного плана миграции.

## Порядок безопасного запуска

1. Ответственный заполняет доказательства и ссылки.
2. Юрист утверждает тексты, основания, сроки и реквизиты оператора.
3. Технический ответственный проводит тесты доступа, аудита, backup/restore и offboarding.
4. В отдельном PR меняется конфигурация gate и прикладывается решение о запуске.
5. CI проверяет полноту конфигурации.
6. После зелёных проверок и ручного production smoke-test разрешается промышленная работа.
7. Любое изменение региона хранения, подрядчика, состава данных или политики доступа повторно закрывает gate до проверки.

Этот документ не является юридическим заключением. Он фиксирует технический запрет на преждевременное включение реальных документов и перечень доказательств, без которых запуск не считается завершённым.
