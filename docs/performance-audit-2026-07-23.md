# Аудит производительности AppСтрой — 23.07.2026

## Масштаб проверки

Проверены 473 Dart-файла, около 89 тысяч строк `lib/`, основная навигация всех ролей, длинные списки, репозитории данных, обработка изображений, realtime-синхронизация, RLS-политики и фактические запросы production Supabase.

Проверка охватила главную, сотрудников и личные данные, табель и отчёты, задачи и фотографии, выплаты и чеки, уведомления, объекты и архив, HR, юридический модуль, управление компаниями и пользователями, панель разработчика, ИИ-диспетчер и служебные центры.

## Найденные причины задержек

1. Тяжёлые вкладки переключались через анимированный `PageView`, одновременно строя два полноценных экрана.
2. Несколько ролей имели отдельные копии неоптимальной оболочки.
3. Длинные списки строили все строки сразу.
4. Профиль и часть форм создавали повторные Future-запросы при rebuild.
5. Открытие задачи ожидало необязательный сетевой запрос до показа редактора.
6. `AppDataSync` синхронно запускал каскад reload/setState после записи.
7. Web-сжатие изображений блокировало UI через `canvas.toDataUrl`.
8. Одинаковые параллельные и принудительные запросы не всегда объединялись.
9. Табель, выплаты, уведомления, сотрудники и задачи многократно вычисляли RLS-контекст для каждой строки.
10. Финансовая карточка главной собирала две суммы через несколько отдельных загрузок.
11. Диспетчерский центр выполнял конфликтующий `INSERT` при каждом обычном чтении.
12. Проверка часового пояса диспетчера сканировала тяжёлое `pg_timezone_names`.
13. Архив и личные данные повторно читались при возвращении на уже открытый экран.
14. В матрице прав отсутствовали покрывающие индексы нескольких внешних ключей.

## Внедрённые оптимизации интерфейса

- мгновенные сохранённые вкладки через ленивый `IndexedStack`;
- единая оптимизированная оболочка руководителя, прораба, HR, бухгалтера, юриста и разработчика;
- сохранение состояния и вложенной навигации уже открытых вкладок;
- отключение ticker-анимаций неактивных экранов;
- мгновенный bottom-tab переход при сохранении свайпов;
- один persistence-write индекса вкладки вместо двух;
- ленивые списки задач, сотрудников, табеля и выплат;
- сохранение позиции прокрутки;
- редактор задачи открывается до завершения необязательной фоновой загрузки;
- асинхронная доставка событий `AppDataSync`.

## Внедрённые оптимизации данных

- кэш профиля, объектов, сотрудников, политик, табеля, выплат, личных данных и архивов;
- in-flight deduplication обычных и принудительных запросов;
- параллельная загрузка независимых данных месячного и периодного табеля;
- параллельная загрузка чанков личных данных;
- один RPC для ленты уведомлений вместе с очисткой и статусами прочтения;
- отдельный быстрый boolean-RPC красной точки;
- set-based RLS-видимость уведомлений;
- быстрые RPC сотрудников, задач и табеля с однократным вычислением объектных прав;
- один RPC выплат вместе с чеками;
- один финансовый агрегат главной вместо цепочки сотрудников → объектов → табеля → выплат;
- удаление холостого `INSERT` из чтения диспетчерского центра;
- встроенная проверка часового пояса вместо сканирования системного представления;
- шесть недостающих индексов внешних ключей матрицы прав.

## Внедрённые оптимизации изображений

- асинхронное web-сжатие через `canvas.toBlob`;
- ограничение размеров декодирования превью;
- устранение лишней base64-копии;
- параллельная загрузка независимых фотографий;
- объединение одинаковых запросов signed URL и метаданных.

## Контрольные production-замеры

Замеры выполнены с реальными JWT и действующими правами компании после применения миграций:

| Контур | Результат после оптимизации |
| --- | ---: |
| Диспетчерский центр | ~13 мс |
| Сохранение настроек диспетчера с validation и audit triggers | ~12 мс |
| Сотрудники, 28 строк | ~19 мс |
| Задачи за день, 6 строк | ~23 мс |
| Табель за период, 429 строк | ~25 мс |
| Финансовая карточка Мурманска за июль | ~20 мс |
| Лента уведомлений | ~66 мс |
| Красная точка уведомлений | ~47 мс плюс 12-секундный клиентский кэш |
| Выплаты и чеки, типичный набор сотрудников | ~65 мс одним запросом |

Крупнейшие подтверждённые изменения:

- диспетчерский центр: примерно 1175 → 13–17 мс;
- сохранение настроек диспетчера: примерно 973 → 12 мс;
- красная точка: примерно 201 → 47–50 мс;
- сотрудники руководителя: примерно 88 → 13–19 мс;
- ежедневные задачи руководителя: примерно 47 → 23 мс;
- финансовая карточка: несколько сетевых запросов заменены одним агрегатом около 20–27 мс.

## Проверка корректности

- наборы сотрудников, задач и выплат сравнивались со старой RLS-выборкой: пропусков и лишних строк не обнаружено;
- объектная изоляция проверялась отдельно для руководителя и прораба;
- права просмотра выплат сохраняют `paid = 0` для роли без соответствующего разрешения;
- очистка, прочтение и персональная видимость уведомлений сохранены;
- после записи соответствующие кэши инвалидируются;
- каждый пакет проходил Flutter analyzer, целевые контракты, полный regression suite, SQL migration guard и web release build;
- итоговый контрольный PR повторно прогоняет полный CI уже на совокупности всех изменений.

## Supabase Performance Advisor

Предупреждений `unindexed_foreign_keys` после миграций не осталось. Advisor показывает только информационные `unused_index`. Эти индексы намеренно не удаляются: статистика после изменений ещё не накопилась, а часть индексов обслуживает редкие операции удаления, аудита и внешние ключи.

## Ограничения

Оптимизация не меняет бизнес-логику, роли, права, расположение навигации, кнопки, дизайн или визуальную структуру экранов. Справочник профессий не добавляется. APK и iOS не собираются; проверяется web release.
