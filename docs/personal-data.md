# Персональные данные

Этот документ описывает технические ограничения приложения. Он не заменяет утверждённые локальные акты и юридическую оценку обработки персональных данных.

## Текущий статус

Промышленная работа с реальными паспортами и другими закрытыми кадровыми документами **не утверждена**.

Обязательный порядок включения и машиночитаемый статус находятся в:

- `docs/personal-data-production-gate.md`;
- `config/personal-data-production-gate.json`.

Пока gate закрыт, используются только тестовые, обезличенные или явно маркированные тестовые копии.

## Категории данных

- ФИО, телефон, должность и объект;
- паспорт, СНИЛС, ИНН и кадровые документы;
- банковские реквизиты;
- фотографии и файлы кандидатов;
- история действий пользователей;
- сведения о выплатах и табеле.

## Минимизация

Каждая роль получает только данные, необходимые для её работы. Открытые рабочие карточки отделяются от закрытых кадровых и банковских данных.

ИИ-функции не получают:

- паспортные реквизиты;
- банковские реквизиты;
- содержимое файлов кандидата;
- закрытые Storage-пути;
- `service_role` ключ.

Закрытые поля подставляются локально после RLS-проверки.

## Согласие кандидата

Пакет документов кандидата нельзя скачать, пока в карточке не подтверждено согласие на обработку персональных данных. Проверка выполняется одновременно:

- в интерфейсе;
- в сервисе формирования пакета.

Отсутствующее согласие нельзя обходить прямым вызовом кнопки или метода клиента.

## Хранение файлов

- buckets с кадровыми документами должны быть закрытыми;
- доступ выдаётся через RLS или краткоживущий разрешённый запрос;
- клиент не доверяет имени файла как пути;
- при формировании ZIP опасные символы и попытки выхода из каталога очищаются;
- тестовые копии не смешиваются с production-документами;
- недоступный файл пропускается с предупреждением, а не заменяется выдуманным содержимым.

Наличие приватного bucket и RLS само по себе не означает, что production-контур разрешён. Должны быть закрыты все доказательства production gate.

## Скачивание и аудит

Скачивание закрытых архивов и документов должно оставлять достаточный аудит: пользователь, компания, кандидат или сотрудник, тип операции и время. Содержимое документа в аудит не записывается.

До подтверждения рабочего журнала скачивания gate остаётся закрытым.

## Удаление и сроки

До включения массового хранения реальных паспортных документов компания должна утвердить:

- цель обработки;
- правовое основание;
- ответственных лиц;
- место хранения;
- сроки хранения;
- порядок архивирования и удаления;
- порядок обработки запросов субъекта;
- действия при инциденте.

Техническое удаление выполняется только после проверки связанных записей и файлов. Для необратимой операции формируется manifest затрагиваемых данных.

## Правило production-включения

`real_personal_documents_enabled` можно изменить на `true` только отдельным PR, когда:

- все поля `evidence` в `config/personal-data-production-gate.json` равны `true`;
- заполнены `approved_by` и `approved_at`;
- приложены проверяемые доказательства;
- утверждены форма согласия, реквизиты оператора, сроки хранения и удаления;
- проверены журнал скачивания, backup/restore и отключение доступа;
- выполнен production release checklist.

CI должен отклонять конфигурацию, которая включает реальные документы без полного набора доказательств.

## Безопасная разработка

- реальные персональные данные не добавляются в тесты, fixtures, логи и PR;
- скриншоты production не публикуются без обезличивания;
- ошибки пользователю не раскрывают SQL, токены и внутренние пути;
- `service_role`, ключи бота и push-секреты хранятся только в защищённом окружении;
- доступ бывшего сотрудника отключается через членство, профиль, устройства и активные сессии;
- до открытия gate реальные паспорта, СНИЛС, ИНН и банковские реквизиты не используются для демонстраций.
